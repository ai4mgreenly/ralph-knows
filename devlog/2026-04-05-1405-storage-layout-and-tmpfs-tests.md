# 2026-04-05 14:05 — storage layout, instance naming, and tmpfs-backed tests

## What

Designed (but did not yet implement) the storage layout for db and socket artifacts, a single-knob dev/prod split, and the integration-test strategy. Verified empirically that fanotify + `open_by_handle_at` work on tmpfs, which was the gating question for the test plan.

## Why these choices

**One canonical instance name, derived from `basename(realpath(watch_path))`.** The watched directory's basename is the natural identity for an instance of ralph-knows: it's what a human would call "the projects index" or "the projects-dev index," and it's unambiguous as long as you canonicalize the path first. Hashing the full path was on the table and rejected — hashes produce illegible filenames (`a3f9c2e1b4d7.db`) and solve a problem the verification layer already solves more cheaply. Collisions between unrelated paths that happen to share a basename (`/mnt/projects` vs `/home/foo/projects`) become a loud, recoverable error at db open rather than silent corruption, which is an acceptable failure mode because it hands the user a clear fix (`--db <name>`).

**Verification on db open: a db file is bound to one canonical `watch_path` for life.** On creation the db stores its `watch_path` in a metadata table; on every subsequent open the current canonical `watch_path` is compared and mismatches are refused with a clear error. This is what makes the basename-derivation scheme safe — whichever naming rule you pick, the real safety comes from verifying the binding at open time. Schema version lives in the same metadata table, same check, same "refuse on mismatch" posture (with migration as the escape hatch later). Creation is transactional: schema + metadata row in one transaction, or the file isn't considered initialized — this prevents a crash mid-init from leaving a "db with no `watch_path` row" that the next open silently adopts.

**Single `RALPH_KNOWS_STATE_DIR` env var as the dev/prod knob.** The current `.envrc` overrides `RALPH_KNOWS_DB_PATH` and `RALPH_KNOWS_SOCKET_PATH` independently, which means any new artifact the config grows needs another hardcoded override, and any change to the derivation scheme bypasses dev entirely. Collapsing to one root directory — `~/.local/state/ralph-knows/` for prod, `~/.local/state/ralph-knows-dev/` for dev — makes dev and prod run identical code paths with the same derivation logic. Everything underneath is computed, nothing is pinned. A single env var change in `.envrc` and dev is a faithful mirror of prod rather than a parallel configuration that can drift.

**`--db` takes a bare filename, not a path.** With state dir handling *where* and watch basename handling *what name*, `--db` only needs to override the derived name within the state dir. Rejecting any value containing `/` keeps the orthogonality clean: `RALPH_KNOWS_STATE_DIR` is the only way to change the storage location, `--db` is the only way to change the name. Option B (smart-flag: filename vs path depending on slash presence) was considered and rejected — the implicit rule is a footgun ("why did `--db foo` and `--db ./foo` do different things") for a convenience we don't need.

**Socket name chains from db name, not directly from watch basename.** The db name is the instance's canonical identifier — it's what `--db` lets you override precisely when the derived basename isn't right. The socket should track the same identifier so that any override drags both artifacts in sync. The alternative (socket rederives from watch basename independently) creates a failure mode where resolving a db collision with `--db projects-home` leaves the socket still colliding, because it didn't hear about the override. Chain it: `watch basename → db name (overridable) → socket name`. One identifier, one override point, two artifacts, zero drift.

**Per-test-file tmpfs mounts for integration tests.** `FAN_MARK_FILESYSTEM` watches an entire filesystem, so test isolation requires a separate filesystem per unit of isolation — subdirectories can't substitute. Options considered: reuse `projects-dev` (rejected — tests would fight active dev work and pollute the running dev instance's index), dedicate a persistent loopback `projects-test` image (workable but introduces a setup ceremony and a persistent artifact), per-run loopback (setup cost + privilege at every test run), tmpfs. tmpfs wins on the ergonomic argument that matters most: **fresh checkouts run `make test` immediately**, with no `make test-setup` ceremony, no fstab entry, no image file to manage. Per-test-file granularity (one mount per test file, tests within a file run serially on that mount) is coarse enough that setup/teardown cost stays negligible but fine enough that file-level parallelism is a clean future upgrade rather than a re-architecture.

**Cleanup via kernel mountinfo + well-known path prefix.** Test mounts land at `/tmp/rk-test-<file>-<pid>/`. On harness startup, sweep `/proc/self/mountinfo` for any mountpoint matching the prefix and unmount it, then `rmdir`. No intent log, no sweep directory, no two-sources-of-truth to reconcile — the kernel already tracks live mounts, and the prefix is how we recognize our own. An intent-file WAL was considered and rejected for cleanup correctness because it duplicates state the kernel maintains; it's still a reasonable future addition if we want debugging metadata beyond "does this mount exist," but cleanup correctness should not couple to it. Aggressive sweep (unconditional unmount of prefix matches) is fine given this is a single-user tool; the pid encoded in the mountpoint name leaves a "paranoid sweep" option available if sharing ever becomes a concern.

**Verified: fanotify + `open_by_handle_at` work on tmpfs (kernel 6.12).** This was the gating question for the whole tmpfs test plan — the watch module depends on resolving fanotify-delivered file handles via `open_by_handle_at` against a mount fd, and tmpfs has historically had uneven exportfs support. We pointed the existing binary at a fresh tmpfs mount, exercised create/modify/rename/delete on files and subdirectories, and confirmed that every event arrived with a fully resolved path — i.e., every handle round-tripped through `open_by_handle_at` successfully. The one `ESTALE` observed was on the tail of a recursive `rm -rf` where the inode was already reclaimed before the handler could resolve it; that's the same race that exists on ext4 and is not tmpfs-specific. The watch module already drops such events with a warning, which is the correct response. tmpfs is cleared to proceed as the integration-test backing store.

## Carrying forward

None of this is built yet. The implementation order that follows from the design:

1. `RALPH_KNOWS_STATE_DIR` env var, with basename-derived db and socket names computed from `realpath(watch_path)`. `--db` becomes a bare-filename override. `.envrc` collapses to a single `RALPH_KNOWS_STATE_DIR=...` line.
2. Verification on db open — metadata table with `watch_path` and schema version, refused on mismatch, created transactionally.
3. Integration test harness: per-test-file tmpfs mounts, per-test-file state dir on the regular filesystem, prefix-based mountinfo sweep on startup, guaranteed unmount on teardown.

Only once (1) and (2) land does the dev-vs-test-vs-prod story actually work end-to-end; the tmpfs verification just confirmed the foundation is solid.
